  • Tailwind CSS + shadcn/ui (design system complet)

 11. DÉTAIL DES MESURES DE SÉCURITÉ IMPLÉMENTÉES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

11.1 — Authentification
  • Email + mot de passe avec confirmation email obligatoire
  • Pas de comptes anonymes
  • Protection contre les mots de passe fuités (leaked password protection)
  • JWT vérifié côté serveur dans TOUTES les Edge Functions
  • userId TOUJOURS extrait du token, jamais du client

11.2 — Row Level Security (RLS)
  • Activé sur toutes les tables
  • Fonction has_role() SECURITY DEFINER pour éviter la récursion
  • Albums : visibilité selon privacy (public/friends/private)
  • Album_media : héritage de la privacy de l'album parent
  • Messages : vérification de participation à la conversation
  • Commandes : visibilité acheteur OU vendeur uniquement
  • security_logs : insertion réservée aux admins
  • Friendships : trigger serveur pour message requests

11.3 — CORS
  • Origines restreintes aux domaines *.lovable.app
  • Header Vary: Origin pour compatibilité CDN
  • Appliqué sur : r2-upload, create-tip, customer-portal, create-checkout,
    _shared/cors.ts (partagé)

11.4 — Rate Limiting
  • Messages : 20/minute (trigger PostgreSQL check_message_rate_limit)
  • Upload R2 : 30/minute par utilisateur
  • Tips : 5/minute par utilisateur
  • IA assistants : 10/minute par utilisateur (seller-ai-coach, ad-assistant)
  • Modération IA : textes < 15 chars ignorés (économie coûts)

11.5 — Validation d'upload (r2-upload)
  • MIME type vérifié (whitelist par dossier)
  • Extension vs MIME : correspondance obligatoire
  • Magic bytes : validation binaire (JPEG, PNG, GIF, WEBP, BMP, HEIC)
  • Taille max par type (avatars: 5MB, vidéos: 50MB, etc.)
  • Fichiers vides rejetés
  • Path traversal bloqué (sanitizePath)
  • Ownership strict (suppression limitée au propre dossier)

11.6 — Stripe & Paiements
  • Prix TOUJOURS récupérés depuis la base de données (jamais du client)
  • create-checkout : priceId validé contre Stripe (prix actif vérifié)
  • Webhook : signature Stripe OBLIGATOIRE (plus de fallback)
  • Tips : montant borné [1€ - 500€] avec trigger DB validate_tip_amount
  • Vérification de stock serveur avant paiement
  • Commissions calculées côté serveur

11.7 — Anti-abus & Trust
  • Trust Score composite (0-100) calculé serveur
  • Facteurs : âge du compte, transactions, social, vérification
  • Empreintes digitales (device fingerprints) pour détection multi-comptes
  • Auto-purge des fingerprints après 90 jours
  • Signalement utilisateurs avec workflow de vérification d'identité
  • Ban utilisateurs / emails / IPs
  • Archives d'usurpation d'identité avec preuves

11.8 — Stockage sécurisé
  • Médias publics → Cloudflare R2 (avec validation serveur)
  • Documents d'identité → Bucket privé Supabase (id-documents)
  • Signed URLs temporaires (5 min) pour accès admin aux documents
  • Backgrounds utilisateur → Bucket privé Supabase (backgrounds)

11.9 — Modération IA
  • Modération automatique des posts et messages
  • Auto-apprentissage via feedback humain (règles apprises stockées en DB)
  • Cache de modération 6h (économie coûts)
  • Modèles IA optimisés par tâche (flash-lite pour simple, flash pour complexe)

 16. DURCISSEMENT SÉCURITÉ — PHASE 2 (8 mars 2026)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Résumé : Audit de sécurité approfondi des Edge Functions avec corrections
appliquées sur 6 fonctions backend. Objectif : uniformiser l'authentification,
le rate limiting, la restriction CORS, et éliminer les usages inutiles de
service_role.

16.1 — Rate Limiting ajouté sur 6 Edge Functions
  • ai-content        → 20 req/min par utilisateur (déjà auth, rate limit ajouté)
  • ai-engine         → 15 req/min par utilisateur
  • post-assistant    → 10 req/min par utilisateur
  • livekit-token     → 10 req/min par utilisateur
  • ad-assistant      → 10 req/min par utilisateur
  • seller-ai-coach   → 10 req/min par utilisateur (renforcé)
  Mécanisme : Map en mémoire par userId, reset toutes les 60 secondes.
  Réponse : HTTP 429 "Trop de requêtes" si dépassement.

16.2 — CORS restreint appliqué uniformément
  • Migration de corsHeaders statiques (Allow-Origin: *) vers getCorsHeaders()
    dynamique sur toutes les fonctions IA et LiveKit.
  • Seuls les domaines *.lovable.app et *.lovableproject.com sont autorisés.
  • Header Vary: Origin inclus pour compatibilité CDN/cache.
  • Fonctions mises à jour : ai-content, ai-engine, livekit-token, seller-ai-coach.

16.3 — Validation de montant renforcée (ad-checkout)
  • Montant validé côté serveur : minimum 1€ (100 centimes), maximum 100 000€.
  • Rejet des montants négatifs, nuls, NaN ou non-numériques.
  • Empêche l'injection de prix frauduleux depuis le client.

16.4 — Réduction des privilèges (check-subscription)
  • Migration de SUPABASE_SERVICE_ROLE_KEY vers SUPABASE_ANON_KEY.
  • La fonction ne nécessitait pas de bypass RLS — seul le JWT est vérifié.
  • Réduit la surface d'attaque en cas de compromission de la fonction.

16.5 — Auth serveur uniformisée
  • Toutes les Edge Functions vérifient désormais le JWT via supabase.auth.getUser()
    AVANT tout traitement.
  • Le userId est TOUJOURS extrait du token serveur, jamais du body client.
  • Réponse uniforme HTTP 401 si auth manquante/invalide.

16.6 — Protection mots de passe fuités
  • Activation de hibp_enabled dans la configuration Auth.
  • Les mots de passe compromis (présents dans les fuites de données connues)
    sont rejetés à l'inscription et au changement de mot de passe.

16.7 — Audit secrets frontend
  • Scan complet du dossier /src : aucune clé privée, aucun service_role_key,
    aucun secret Stripe détecté côté client.
  • Seules les clés publiques (VITE_SUPABASE_PUBLISHABLE_KEY, anon key) sont
    présentes — conforme aux bonnes pratiques.

16.8 — Résumé des fichiers modifiés
  • supabase/functions/ai-content/index.ts      — rate limit + CORS restreint
  • supabase/functions/ai-engine/index.ts        — rate limit + CORS restreint
  • supabase/functions/post-assistant/index.ts   — rate limit + auth renforcée
  • supabase/functions/livekit-token/index.ts    — rate limit + CORS restreint
  • supabase/functions/ad-checkout/index.ts      — validation montant serveur
  • supabase/functions/check-subscription/index.ts — downgrade vers anon key
